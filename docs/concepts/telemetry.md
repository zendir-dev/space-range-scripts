# Telemetry

Telemetry is everything flowing **from** a spacecraft **to** a team — periodic status, captured imagery, schedule reports, intercepted foreign uplinks. All of it travels the same path: generated by the spacecraft, encoded as a CCSDS Space Packet (or media/intercept frame), passed through the simulated RF link, and finally published on the team's `Downlink` MQTT topic.

This page explains how that path is built, how clients should decode it, and how the RF link budget can drop telemetry that would otherwise be delivered.

---

## The downlink path, end-to-end

```text
spacecraft component
        │  (generates payload, e.g. a Ping struct)
        ▼
  CCSDS Space Packet construction
        │  (packet headers + XTCE-typed payload bytes)
        ▼
  Caesar cipher (key = team Caesar key)
        │
        ▼
  per-team "downlink buffer"
        │
        ▼
  Transmitter.TransmitBytes()
        │  (RF link budget evaluated against ground stations)
        ▼
  if link closes → emit on MQTT       if link fails → bytes are dropped
        │
        ▼
  prepend 5-byte frame header (Format + TeamID)
        │
        ▼
  XOR encryption (key = team password)
        │
        ▼
publish on Zendir/SpaceRange/<GAME>/<TEAM>/Downlink
```

Decoding on the client is the reverse:

1. Receive the MQTT payload.
2. **XOR-decrypt** with the team password.
3. Split off the **5-byte frame header**: 1 byte `Format`, 4 bytes little-endian `TeamID`.
4. **Caesar-decrypt** the remaining bytes with the team's Caesar key.
5. Parse the result according to the `Format` field, using the team's **XTCE schema** for `Message` payloads.

A worked example with code is in [Guides → Decoding telemetry](../guides/decoding-telemetry.md).

## The 5-byte frame header

Every downlink packet, regardless of payload type, has a 5-byte header (between the XOR layer and the Caesar layer):

| Offset | Size | Field | Meaning |
| --- | --- | --- | --- |
| 0 | 1 byte | `Format` | `EDataFormatType`: `0` = None, `1` = Message, `2` = Media, `3` = Uplink Intercept |
| 1 | 4 bytes | `TeamID` | Little-endian 32-bit integer. The team that owns the spacecraft that emitted this packet. |
| 5 | … | Payload | Payload bytes (Caesar-encoded). |

`TeamID` is included so a client receiving traffic on a topic it doesn't fully control can still tell *which* team's spacecraft a packet came from. In practice, this is also a sanity check — receiving a frame whose `TeamID` doesn't match your configured team is a sign that someone has crossed wires.

The four format codes correspond to four payload shapes; each is described below.

---

## Format = `Message` (CCSDS Space Packets)

The most common downlink format. The payload is a **CCSDS Space Packet** with primary header per CCSDS 133.0-B-2 and a typed data field whose layout is described by the team's XTCE schema.

Space Range emits two distinct Message types out of the box:

| Message | When it's sent | Schema name |
| --- | --- | --- |
| **Ping** | Periodically (cadence is a per-spacecraft setting), and on demand via the `downlink` command with `ping=true`. | `Ping` |
| **Schedule Report** | In response to the [`get_schedule`](../api-reference/spacecraft-commands.md#get_schedule) command. | `ScheduleReport` |

Both ride the same path and are demultiplexed on the client side using the **APID** field of the Space Packet primary header. The XTCE schema tells you which APID corresponds to which message.

### Ping

A Ping is a snapshot of the spacecraft's current health and recent activity. Fields:

| Field | Type | Description |
| --- | --- | --- |
| `State` | string | Current operational state: `NOMINAL`, `LOW`, `SAFE`, `TRANSMIT`, `FULL STORAGE`, or `REBOOTING`. |
| `Station` | string | Name of the nearest ground station, or `None` if not in range. |
| `Memory` | float (0–1) | On-board storage usage as a fraction of capacity. |
| `Battery` | float (0–1) | Battery charge as a fraction of capacity. |
| `Commands` | string (JSON) | Commands executed since the previous Ping, as a JSON array. Each entry has `ID`, `Command`, `Time`, `Success`, `Args` (with `password` redacted). Defaults to `"[]"`. |
| `UplinkInterceptDataBytes` | int (bytes) | Total bytes of intercepted uplink frames currently held in the on-board ring buffer awaiting downlink. |

Ping is the primary mechanism for confirming command execution. Watch the `Commands` array for entries matching the `(Time, Command)` of an uplink you sent.

### Schedule Report

A Schedule Report is a one-shot dump of the spacecraft's current pending command queue. Fields:

| Field | Type | Description |
| --- | --- | --- |
| `Count` | int | Number of pending commands. |
| `Commands` | string (JSON) | Pending commands as a JSON array. Each entry has `Asset`, `ID`, `Time`, `Command`, `Args` (redacted). Defaults to `"[]"`. |

A Schedule Report is generated only in direct response to [`get_schedule`](../api-reference/spacecraft-commands.md#get_schedule). It is not periodic.

> **The `Commands` field is a JSON-encoded string, not a JSON array.** Both Ping and Schedule Report deliver their command lists as a string field whose contents are valid JSON. After XTCE-parsing the packet, you must call `JSON.parse(packet.Commands)` (or equivalent) to access the array. This is a deliberate choice that keeps the XTCE schema fixed-size and lets the JSON grow.

The full binary layouts are in [Reference → Packet formats](../reference/packet-formats.md).

## Format = `Media` (imagery and files)

When a camera captures an image or a media file is downlinked, the payload is wrapped as Media:

| Offset | Size | Field |
| --- | --- | --- |
| 0 | 50 bytes | File name, padded with `\0` to 50 bytes. UTF-8 / ASCII. |
| 50 | … | Raw file bytes (e.g. JPEG/PNG image). |

The 50-byte name header is fixed-width — extract bytes 0..50, strip trailing nulls, and decode as UTF-8 to get the file name; everything from byte 50 onward is the file body.

Imagery is corrupted probabilistically before storage and downlink to model real-world data integrity issues; corruption flips bits in a randomly selected fraction of the bytes.

## Format = `Uplink Intercept`

When a spacecraft has uplink-intercept recording enabled, the receiver captures **every** uplink it sees on its frequency — including foreign uplinks intended for other teams' spacecraft — and emits them on downlink as Uplink Intercept records.

The payload is a fixed 32-byte header followed by up to 1024 bytes of raw on-air ciphertext. Header layout:

| Offset | Size | Field | Description |
| --- | --- | --- | --- |
| 0 | 4 bytes | `Magic` | Constant `0x5055495A` ("ZIUP" in little-endian). |
| 4 | 1 byte | `Version` | Format version. Currently `2`. |
| 5 | 3 bytes | (padding) | Zero. |
| 8 | 8 bytes | `SimTimeSeconds` | Double, simulation time when the frame was dequeued. |
| 16 | 4 bytes | `WireLengthBytes` | Original on-air length of the frame (may exceed the stored length). |
| 20 | 4 bytes | `StoredLengthBytes` | Bytes actually stored in this record (= `min(WireLengthBytes, 1024)`). |
| 24 | 1 byte | `Flags` | Bitmask: `DecodeOk`, `ParseOk`, `AddressedToUs`, `Truncated`. |
| 25 | 3 bytes | (padding) | Zero. |
| 28 | 4 bytes | `ReceiverFrequency` | Float (LE), receive frequency in MHz. |
| 32 | … | Payload | Up to 1024 bytes of raw ciphertext (start of the original frame). |

Uplink Intercept is a SIGINT training aid:

- **Foreign frames** are stored as on-air ciphertext (i.e. re-encrypted with the receiver's key model so replay is byte-identical).
- **Own-team frames** appear with the `AddressedToUs` flag set so analysts can filter them out.
- Trying to **replay** a stored frame on a different receiver requires it to share the same crypto and frequency as the original target, which is the same problem the original sender faced.

A full walkthrough of decoding intercepts (and the JS parser used by the Operator UI) is in [Guides → Decoding telemetry](../guides/decoding-telemetry.md#uplink-intercepts).

## XTCE schemas

Studio describes its Space Packet payloads using **XTCE** — the CCSDS XML standard for telemetry definitions. Each team has its own XTCE document, retrievable via the ground-controller request:

```python
ground.request("get_packet_schemas")
# → "args": { "schemas": [
#     { "name": "Ping",           "xtce": "<?xml ...?>" },
#     { "name": "ScheduleReport", "xtce": "<?xml ...?>" },
#     ...
#   ]}
```

The XTCE document gives you, for each Space Packet:

- The APID (so you can route incoming packets by header).
- The fields of the data section, with names, types, units, ranges, and descriptions.
- Enumerations and string lengths where relevant.

A custom client doesn't have to fully parse XTCE — it can use the field metadata to interpret the bytes following the primary header. For convenience, the `space-range-scripts` Python package and the Operator UI both ship with a tiny XTCE-aware parser that converts a raw Space Packet into a typed dictionary.

You should re-fetch schemas whenever the [instance ID](simulation-clock.md#instance-id-and-resets) changes, in case the new scenario uses different definitions.

## The RF link budget

Telemetry is **not** guaranteed to arrive. Every time a spacecraft transmits, Studio computes a per-receiver link budget that decides whether the packet closes the link to the team's ground network. Inputs:

- Spacecraft transmitter power and antenna gain.
- Ground station transmitter power, antenna gain, and bandwidth.
- Range (line-of-sight distance from spacecraft to nearest receiving station).
- Frequency.
- Active **jamming** by other spacecraft on overlapping frequencies.
- Any stochastic margin baked into the simulation.

If the budget closes, the bytes are emitted on the MQTT downlink path. If it doesn't, the packet is silently dropped. From the operator's perspective the spacecraft is "off the air" — no error, no notification, just silence on `Downlink`.

You can fetch the most recent uplink/downlink budget snapshots from the ground controller via [`get_telemetry`](../api-reference/ground-requests.md#get_telemetry):

```python
ground.request("get_telemetry", {"asset_id": "A3F2C014"})
# → "args": {
#     "asset_id": "A3F2C014",
#     "frequency": 2230.0,
#     "key": 17,
#     "bandwidth": 1.0e6,
#     "uplink":   { "snr_db": ..., "link_margin_db": ..., ... },
#     "downlink": { "snr_db": ..., "link_margin_db": ..., ... }
#   }
```

(Whichever fields a budget includes depend on the simulation's link-budget model. The shape is the same `LinkMessage` struct used internally; treat all fields as optional from the client's perspective.)

### Why telemetry might be missing

A short checklist for "I'm not getting Pings":

| Cause | What to check |
| --- | --- |
| Spacecraft is shut down (e.g. just rotated keys) | Wait for the reset interval; expect `REBOOTING` state on next Ping. |
| Frequency or Caesar key drift between spacecraft and ground | [`get_telemetry`](../api-reference/ground-requests.md#get_telemetry) and confirm both ends agree. |
| Out of view | Check spacecraft position vs. ground station latitudes; a polar ground station won't see equatorial passes. |
| Active jammer | List entities and check for jammers with `is_active: true` overlapping your frequency. |
| Wrong XTCE schema (post-reset) | Re-fetch with [`get_packet_schemas`](../api-reference/ground-requests.md#get_packet_schemas). |
| XOR password mismatch | Confirm you're using the team password, not the admin password. |

## Storage and `downlink`

Most telemetry is *not* automatically transmitted on every tick — the spacecraft writes it into an **on-board storage buffer** and transmits when the [`downlink`](../api-reference/spacecraft-commands.md#downlink) command tells it to (or, for Pings, on the configured periodic cadence). Storage has a finite capacity; once full, new payloads are dropped until storage is downlinked or cleared. Watch the Ping `Memory` field to anticipate this.

The `downlink` command also has a `ping` flag that forces an immediate Ping in addition to flushing storage — useful when you've just changed something and want a fresh status without waiting for the next cadence.
